\section{Approche pour l'implémentation du jeu}
\label{sec:part-salah}


\subsection{Représentation du plateau et choix d’implémentation}

\subsubsection{Pourquoi un graphe ?}

Le plateau du jeu Coursidor est modélisé comme un graphe, où chaque case correspond à un sommet, et chaque possibilité de déplacement à une arête. Ce choix s’impose naturellement dans un jeu où les déplacements doivent être validés selon des règles de voisinage, notamment en présence d'obstacles (murs).

Représenter le plateau sous forme de graphe nous a permis :
\begin{itemize}
    \item d’implémenter plusieurs types et architectures de plateaux (hexagonal, troué, cyclique) à partir d’une même structure générique.
    \item de maintenir un code modulaire et réutilisable, indépendant du type de graphe utilisé.
    \item de préparer l’implémentation future d’algorithmes d’exploration plus complexes, comme ceux de plus court chemin (par exemple Dijkstra ou A*), sans nécessiter de réécriture fondamentale.
\end{itemize}

\subsubsection{Pourquoi GSL ?}

Le choix de l'utililsation des matrices creuses au format compressé (CSR) fournies par la bibliothèque \texttt{GSL} (\texttt{gsl\_spmatrix\_uint}) pour représenter la matrice d’adjacence du graphe s’appuie sur plusieurs avantages essentiels pour notre projet :

\begin{itemize}
    \item la majorité des sommets n’ont que quelques voisins (au plus 6 dans une grille hexagonale), ce qui rend la matrice d’adjacence très creuse.
    \item le format CSR permet de stocker uniquement les entrées non nulles, réduisant considérablement l’espace mémoire nécessaire, ce qui est crucial sur des plateaux de grande taille.
    \item GSL fournit des fonctions efficaces pour itérer sur les voisins d’un sommet sans parcourir toute la matrice, ce qui améliore les performances des fonctions de génération et de validation des mouvements.
    \item l’utilisation d’une structure creuse rend explicite la topologie du graphe tout en maintenant une bonne flexibilité pour les manipulations (ajout, blocage, suppression d’arêtes).
\end{itemize}

Nous avons adopté une codification explicite des arêtes dans la matrice, inspirée de l’énumération suivante :

\begin{itemize}
    \item \texttt{0} (\texttt{NO\_EDGE}) : aucune arête n’existe entre les deux sommets (non connectés) ;
    \item \texttt{7} (\texttt{WALL\_DIR}) : l’arête est bloquée par un mur et donc inutilisable ;
    \item \texttt{1} à \texttt{6} : indication de la direction de l’arête selon le repère hexagonal, à savoir :
    \begin{itemize}
        \item \texttt{1 (NW)} : nord-ouest  
        \item \texttt{2 (NE)} : nord-est  
        \item \texttt{3 (E)}  : est  
        \item \texttt{4 (SE)} : sud-est  
        \item \texttt{5 (SW)} : sud-ouest  
        \item \texttt{6 (W)}  : ouest  
    \end{itemize}
\end{itemize}

\subsubsection{Coordonnées axiales}
Afin de représenter efficacement un plateau hexagonal, nous avons implémenté un système de coordonnées dit \textbf{axial}, très utilisé dans les graphes hexagonaux. Chaque case est identifiée par un couple $(l, c)$, correspondant aux coordonnées « ligne » et « colonne » dans ce système.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.6\textwidth]{rapport/axial.png}
    \caption{Graphe hexagonal de rayon \(m=3\) avec coordonnées axiales décalées}
    \label{fig:graph-m2}
  \end{figure}

\paragraph{Type \texttt{axial\_t}}
Le type de données \texttt{axial\_t} est une simple structure contenant deux entiers :

\begin{lstlisting}[language=C]
    struct axial_t {
        int l; // ligne
        int c; // colonne
    };
    \end{lstlisting}
    
Ce type est utilisé dans la majorité des fonctions de construction du graphe pour :
\begin{itemize}
    \item vérifier simplement l’appartenance d’un sommet au graphe à partir de ses coordonnées axiales.
    \item déterminer et parcourir les voisins d’un sommet selon les 6 directions définies, en appliquant des décalages sur \((l,c)\).
    \item convertir un couple de coordonnées axiales en un indice unique dans la matrice d’adjacence selon le type du graphe grâce à une fonction \texttt{\texttt{int} axial\_to\_index(\texttt{int} l, \texttt{int} c, \texttt{int} m, \texttt{int} type)}.
\end{itemize}

\subsubsection{Implémentation du graphe}
Le graphe est généré dans la fonction \texttt{graph\_generate()}, qui :
\begin{itemize}
    \item parcourt chaque coordonnée \texttt{axial\_t} du plateau ;
    \item vérifie si la coordonnée est valide avec une fonction \texttt{in\_hexagon()...} ;
    \item assigne un index unique à chaque sommet (via \texttt{axial\_to\_index()}) ;
    \item ajoute des arêtes entre sommets voisins dans les 6 directions si elles sont valides.
\end{itemize}

Cela permet de générer dynamiquement plusieurs types de plateaux selon les besoins.

\subsection{Chargement dynamique des stratégies}

Le fichier \texttt{server.c} accepte deux bibliothèques dynamiques (\texttt{.so}) passées en argument à l’exécution. Le serveur ouvre chacune d’elles avec \texttt{dlopen()} et récupère les fonctions standard (\texttt{get\_player\_name}, \texttt{initialize}, \texttt{play}, \texttt{finalize}) via \texttt{dlsym()}, sans qu’il soit nécessaire de modifier ou de recompiler le code du serveur.

\begin{itemize}
  \item \textbf{Flexibilité} : au lancement, on change simplement les noms des bibliothèques passées en paramètre de l’exécutable pour tester différentes stratégies.
  \item \textbf{Modularité} : chaque stratégie est empaquetée dans sa propre bibliothèque, facilitant le développement, la maintenance et les tests isolés.
  \item \textbf{Chargement} : un pointeur \texttt{library} est initialisé avec le résultat de \texttt{dlopen()} pour lier dynamiquement les symboles.
\end{itemize}

\begin{lstlisting}[language=C]
struct player_t {
    enum player_color_t     player_color;
    char const *(*get_player_name)();
    void (*initialize)(unsigned int, struct graph_t *);
    struct move_t (*play)(const struct move_t);
    void (*finalize)();
    void *library;        // pointeur vers la bibliotheque dynamique
    vertex_t pos_actuel;
};
\end{lstlisting}

Une fonction \texttt{load\_player()} a pour unique rôle de charger, au démarrage de la partie, les deux bibliothèques dynamiques correspondant aux stratégies des joueurs. En appelant successivement \texttt{dlopen()} puis \texttt{dlsym()} pour chaque fonction standard des joueurs.
